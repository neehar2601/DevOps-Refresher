Docker Networking Overview

Docker networking enables containers to communicate with each other, the Docker host, and external networks. Docker uses the Container Network Model (CNM) to standardize networking across different environments.

Key Networking Concepts

Network Namespaces

Each container runs in its own network namespace, providing isolation of network stack, IP addresses, and routing tables.

Virtual Ethernet (veth) Pairs

Docker uses veth pairs to connect containers to networks. One end is inside the container, the other connects to the host.

Bridges

Act as virtual switches that forward traffic between containers and the host network.

Port Mapping

Containers expose specific ports to the host using port mapping (-p flag), enabling external access.

Service Discovery

Containers on the same network can discover each other using container names via built-in DNS.

Network Address Translation (NAT)

Docker uses NAT to route traffic between containers and external networks through the host.

Bridge Network

The bridge network is Docker's default networking mode. It creates a private internal network on the host, allowing containers on the same bridge to communicate while providing isolation from the host and external networks.

Key Feature: Automatic service discovery via DNS - containers can reach each other using container names.

How It Works

1. Bridge Creation

Docker creates a virtual bridge (docker0) on the host, acting as a software switch.

2. Virtual Ethernet Pairs

Each container gets a veth pair - one end inside container (eth0), other end attached to docker0.

3. IP Assignment

Containers receive private IPs from the bridge subnet (default: 172.17.0.0/16).

4. Internal Communication

Containers communicate directly via the bridge using IPs or container names.

5. NAT for External Access

Docker uses iptables NAT rules to route external traffic through the host.

6. Port Publishing

Use -p flag to map container ports to host ports for external access.

  • Easy to set up - default configuration
  • Automatic network isolation
  • Built-in DNS service discovery
  • Suitable for single-host deployments
  • Low overhead and good performance

‚úó Limitations

  • Limited to single host
  • Requires port mapping for external access
  • No multi-host service discovery
  • NAT introduces slight latency
  • Not suitable for distributed applications

Usage Examples

# Use default bridge
docker run -d --name nginx nginx

# Create custom bridge network
docker network create --driver bridge my-bridge

# Run container on custom bridge
docker run -d --name app --network my-bridge nginx

# Publish ports to host
docker run -d -p 8080:80 --name web nginx

# Inspect bridge network
docker network inspect bridge

Host Network

The host network mode removes network isolation between the container and the Docker host. Containers share the host's network stack directly, using the same IP addresses and ports.

Warning: No network isolation means containers can access all host network interfaces and ports. Use carefully!

How It Works

1. No Network Namespace

Container doesn't get its own network namespace - uses host's directly.

2. Direct Host Access

Container sees all host network interfaces (eth0, lo, etc.) as if running on host.

3. No Port Mapping

-p flag has no effect. Container listens on host ports directly.

4. Same IP Address

Container uses the host's IP address for all network operations.

5. Performance Benefits

Eliminates network virtualization overhead - best performance.

6. Port Conflicts

Multiple containers can't bind to the same port - need to coordinate.

‚úì Advantages

  • Maximum network performance
  • No NAT overhead or latency
  • Direct access to host network services
  • Simpler configuration for some scenarios
  • Suitable for network monitoring tools

‚úó Limitations

  • No network isolation or security
  • Port conflicts between containers
  • Less portable across environments
  • Not available on Docker Desktop for Mac/Windows
  • Can't use with Docker Swarm services

Use Cases

Performance-Critical Apps

Applications requiring maximum network throughput with minimal latency.

Network Monitoring

Tools that need to monitor all network traffic on the host.

System Services

Services that need to integrate tightly with host networking (VPNs, proxies).

Development Testing

Quick testing where isolation isn't needed and performance matters.

Usage Examples

# Run container with host network
docker run -d --network host --name nginx nginx

# Container listens on host's port 80 directly
# No -p flag needed or allowed

# Check container network mode
docker inspect nginx | grep NetworkMode

# Example: Network monitoring tool
docker run -d --network host --name monitor nicolaka/netshoot

Overlay Network

Overlay networks enable multi-host networking for Docker Swarm and distributed applications. They create a virtual network spanning multiple Docker hosts, allowing seamless container communication across hosts.

Key Feature: Built-in encryption support for secure communication between hosts and automatic service discovery across the cluster.

How It Works

1. VXLAN Encapsulation

Uses VXLAN (Virtual Extensible LAN) to encapsulate container traffic over the physical network.

2. Multi-Host Spanning

Creates a single logical network across multiple Docker hosts in a swarm.

3. Automatic Routing

Docker handles routing packets between hosts transparently.

4. Service Discovery

Built-in DNS allows containers to discover services across the entire cluster.

5. Encryption Support

Optional IPsec encryption for secure communication between hosts.

6. Load Balancing

Ingress network provides load balancing for published service ports.

  • Automatic load balancing (ingress)
  • Optional encryption for security
  • Seamless with Docker Swarm
  • Scalable for distributed applications

‚úó Limitations

  • Requires Docker Swarm mode
  • More complex setup than bridge
  • Slight performance overhead from encapsulation
  • Requires open ports between hosts
  • Encryption adds additional overhead

Usage Examples

# Initialize Docker Swarm
docker swarm init

# Create overlay network
docker network create --driver overlay my-overlay

# Create service on overlay network
docker service create --name web --network my-overlay --replicas 3 nginx

docker network create --driver overlay --opt encrypted my-secure-overlay

# List overlay networks
docker network ls --filter driver=overlay

# Inspect overlay network
docker network inspect my-overlay

Macvlan Network

Macvlan networks assign unique MAC addresses to containers, making them appear as physical devices on the network. This allows containers to be directly connected to the physical network without NAT or port mapping.

Use Case: Legacy applications that expect physical network interfaces or integration with network appliances requiring MAC addresses.

How It Works

1. Unique MAC Address

Each container receives its own MAC address, separate from the host.

2. Physical Network Integration

Containers appear as physical devices on the network to external systems.

3. Direct Network Access

No NAT or port mapping - containers have direct IPs on physical network.

4. VLAN Support

Can assign containers to specific VLANs (802.1Q tagging).

5. No Docker Bridge

Bypasses Docker's virtual bridge entirely for maximum performance.

6. Parent Interface

Macvlan networks must be bound to a physical host interface.

‚úì Advantages

  • Containers appear as physical devices
  • No NAT overhead - better performance
  • Direct physical network integration
  • VLAN support for network segmentation
  • Works with legacy network infrastructure

‚úó Limitations

  • Requires promiscuous mode on host NIC
  • May not work in cloud environments
  • More complex IP management
  • Limited by MAC address table size on switches
  • Containers can't reach host by default

Macvlan Modes

Bridge Mode (Default)

Containers on same host can communicate. Traffic to external network goes through parent interface.

802.1q Trunk Mode

VLAN tagging support - assign containers to specific VLANs on trunked interface.

Usage Examples

# Create macvlan network
docker network create -d macvlan \
--subnet=192.168.1.0/24 \
--gateway=192.168.1.1 \
-o parent=eth0 my-macvlan

# Run container on macvlan
docker run -d --network my-macvlan --ip=192.168.1.100 --name web nginx

# Create macvlan with VLAN (802.1q)
docker network create -d macvlan \
--subnet=192.168.10.0/24 \
-o parent=eth0.10 vlan10

# Inspect macvlan network
docker network inspect my-macvlan

IPvlan Alternative

IPvlan is similar to Macvlan but uses the parent interface's MAC address instead of assigning unique MACs. This is more efficient for high-density environments where MAC address table limits are a concern.

  • Network monitoring tools
  • Direct host network access required

  • Docker Swarm deployments
  • Multi-host communication needed
