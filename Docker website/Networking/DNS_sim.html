Recursive — How it works

  1. Client sends one query to the Recursive Resolver
  2. Resolver queries Root → gets referral to TLD
  3. Resolver queries TLD → gets referral to Auth NS
  4. Resolver queries Auth NS → gets the final IP
  5. Resolver caches & returns IP to the Client
Client burden: minimal — one query, one answer.

Iterative — How it works

  1. Client queries Root Server directly
  2. Root returns a referral to the TLD server
  3. Client queries TLD Server directly
  4. TLD returns a referral to the Auth NS
  5. Client queries Auth NS → gets the final IP
Client burden: high — must follow every hop.
R Recursive Query

Client asks Resolver: "Get me the answer." The resolver does all the work and returns the final IP.

I Internal Iterative

The resolver itself uses iterative queries (Root→TLD→Auth) internally while the client waits.

C Caching

The resolver caches the result for the record's TTL, skipping the full hierarchy on repeat queries.

Q Direct Query

The client sends each DNS query itself — starting with the Root Server, no resolver in the middle.

↩ Referral Hops

Each server returns the address of the next server to ask — the client follows each referral itself.

✓ Final Answer

Only the Authoritative NS returns the real IP. The client collected all the referrals itself along the way.
