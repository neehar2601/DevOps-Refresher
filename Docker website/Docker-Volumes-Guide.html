Docker Storage Overview

By default, container data is stored in the container's writable layer and lost when the container is removed. Docker provides three mount types for persistent storage: volumes, bind mounts, and tmpfs mounts.

Best Practice: Use volumes for production data persistence. They're managed by Docker, work across platforms, and offer better performance than bind mounts on Docker Desktop.

Key Concepts

Persistence

Data survives container restarts and deletions when stored in volumes or bind mounts.

Isolation

Container storage layer is isolated from volumes - data doesn't count toward container size.

Sharing

Multiple containers can mount the same volume simultaneously for data sharing.

Performance

Volumes offer native disk performance. tmpfs offers RAM speed. Bind mounts vary by platform.

Management

Docker CLI commands manage volumes. Host OS manages bind mount directories.

Portability

Volumes work consistently across Docker hosts. Bind mounts depend on host filesystem structure.

Docker Volumes

Volumes are the preferred mechanism for persisting data in Docker. They're completely managed by Docker, stored in a dedicated directory on the host (/var/lib/docker/volumes/ on Linux), and isolated from core host functionality.

Recommendation: Always use volumes for production databases, application data, and any content that must persist beyond container lifecycle.

How Volumes Work

1. Volume Creation

Created explicitly with docker volume create or automatically when container starts with -v flag.

2. Docker Management

Docker handles storage location, permissions, and lifecycle independently of containers.

3. Mount Point

Volume mounted to specific directory inside container. Multiple containers can share same volume.

4. Data Persistence

Data remains even after container deletion. Must explicitly run docker volume rm to delete.

5. Named vs Anonymous

Named volumes have user-defined names. Anonymous volumes get random UUIDs.

6. Pre-population

If mount point contains data in image, that data is copied to new volume on first mount.

Volume Types

Named Volumes

  • User-friendly names (my-app-data)
  • Easy to reference and manage
  • Ideal for production use
  • Can be shared between containers
  • Created with: docker volume create name

Anonymous Volumes

  • Auto-generated UUID names
  • Created implicitly with -v flag
  • Harder to track and manage
  • Useful for temporary data
  • Often orphaned after container removal

Bind Mounts

Bind mounts link a specific file or directory on the host to a container location. They provide direct, immediate synchronization between host and container filesystems. Ideal for development workflows where you need real-time code updates without rebuilding images.

Security Note: Bind mounts give containers direct access to host filesystem. This can be a security risk. Use carefully in production.

How Bind Mounts Work

1. Direct Mapping

Specific host path mapped directly to container path. Must use absolute paths.

2. Immediate Sync

Changes on host appear instantly in container and vice versa. No copy operation.

3. Host Dependency

Requires exact path to exist on host. Not managed by Docker.

4. File Permissions

Host file permissions apply. Container processes must have appropriate access rights.

5. Overwrites Container Data

If mount path contains data in image, host directory content takes precedence.

6. Platform Differences

Performance varies by platform. Slower on Docker Desktop (Mac/Windows) due to filesystem translation.

Advantages & Limitations

âœ“ Advantages

  • Direct, real-time file access
  • Perfect for development workflows
  • Easy to inspect and edit files
  • No Docker commands needed to access data
  • Can mount specific files (not just dirs)
  • Useful for config file injection

âœ— Limitations

  • Reduced portability - host-dependent paths
  • Security risk - direct host access
  • Performance issues on Mac/Windows
  • Manual path management required
  • Can break container isolation
  • Not recommended for production

Usage Examples

# Basic bind mount (short syntax)
docker run -d -v /host/path:/container/path nginx

-v $(pwd)/src:/app/src \
-p 3000:3000 \

-v /home/user/nginx.conf:/etc/nginx/nginx.conf:ro \

# Using --mount (long syntax, more explicit)

# Read-only bind mount
docker run -d -v /host/config:/app/config:ro nginx

# Mount with propagation mode
docker run -d \
--mount type=bind,source=/host/data,target=/data,bind-propagation=shared \

tmpfs Mounts

tmpfs mounts store data in the host system's memory (RAM) only. They provide the fastest I/O performance but are temporary - all data is lost when the container stops. Perfect for sensitive information that shouldn't be persisted to disk or temporary caches requiring maximum speed.

Linux Only: tmpfs mounts are only available on Linux hosts. Not supported on Docker Desktop for Mac or Windows.

How tmpfs Works

1. Memory Storage

Data stored entirely in host's RAM. No disk I/O - fastest possible performance.

2. Temporary Nature

All data automatically deleted when container stops. Perfect for temporary files.

3. Size Limits

Limited by available system memory. Can set explicit size limits to prevent OOM issues.

4. No Persistence

Zero disk footprint. Data never written to storage layer. Automatic cleanup.

5. Security Benefits

Sensitive data never hits disk. Reduces risk of data leaks from disk forensics.

6. Platform Limitation

Linux only. Uses native tmpfs filesystem. Not available on Docker Desktop.

Advantages & Limitations

âœ“ Advantages

  • Extremely fast - RAM speed I/O
  • Automatic cleanup on container stop
  • Ideal for sensitive data (passwords, keys)
  • No disk space consumption
  • Better security - data never on disk
  • Perfect for temporary caches

âœ— Limitations

  • Linux hosts only (no Mac/Windows)
  • Data lost on container stop
  • Limited by available RAM
  • Not suitable for persistent data
  • Can cause OOM if overused
  • Can't be shared between hosts

Usage Examples

# Basic tmpfs mount (short syntax)
docker run -d --tmpfs /app/cache nginx

# tmpfs with size limit (default unlimited)
docker run -d --tmpfs /tmp:rw,size=100m,mode=1777 nginx

# Using --mount (long syntax, more options)
docker run -d \
--mount type=tmpfs,destination=/app/tmp,tmpfs-size=104857600 \
ubuntu

# tmpfs with specific mode (permissions)
docker run -d \
--mount type=tmpfs,destination=/run/secrets,tmpfs-mode=1700 \
alpine

# Multiple tmpfs mounts
docker run -d \
--tmpfs /tmp \
--tmpfs /app/cache:rw,size=50m \
nginx

# Verify tmpfs mount
docker exec container-name df -h /app/cache

tmpfs Options

Option Description Example
tmpfs-size Size limit in bytes (default: unlimited) tmpfs-size=104857600 (100MB)
tmpfs-mode File mode in octal (default: 1777) tmpfs-mode=1700 (owner only)

Usage Examples

# Default local driver (implicit)
docker volume create my-volume

# Explicit local driver
docker volume create --driver local my-local-volume

# NFS volume using local driver
docker volume create --driver local \
--opt type=nfs \
--opt o=addr=192.168.1.100,rw \
--opt device=:/path/to/dir \
nfs-volume

# CIFS/SMB volume
docker volume create --driver local \
--opt type=cifs \
--opt device=//server/share \
--opt o=username=user,password=pass \
smb-volume

# AWS EBS with RexRay driver
docker volume create --driver rexray/ebs \
--name my-ebs-volume \
--opt size=10

# Run container with driver-based volume
docker run -d -v my-ebs-volume:/data nginx

# List volumes by driver
docker volume ls --filter driver=local

Volume Backup & Restore

Backing up Docker volumes is critical for disaster recovery, data migration, and testing. Docker volumes can be backed up by creating temporary containers that mount the volume and use tar or application-specific tools to create archives.

Best Practice: For databases, use application-specific backup tools (pg_dump, mysqldump, mongodump) for consistency guarantees. Generic tar backups may capture inconsistent state.

Backup Methods Comparison

Method Pros Cons Best For

Generic Volume Backup (tar method)

# Backup volume to tar file
docker run --rm \
-v my-volume:/data \
-v $(pwd):/backup \
ubuntu tar czf /backup/my-volume-backup.tar.gz -C /data .

# Alternative: Stop container first for consistency
docker stop my-app
docker run --rm -v my-volume:/data -v $(pwd):/backup \
ubuntu tar czf /backup/my-volume-backup.tar.gz -C /data .
docker start my-app

# Restore volume from tar backup
docker run --rm \
-v my-volume:/data \
-v $(pwd):/backup \
ubuntu bash -c "cd /data && tar xzf /backup/my-volume-backup.tar.gz"

# Verify backup file
tar tzf my-volume-backup.tar.gz | head

Database-Specific Backups

# PostgreSQL backup
docker exec postgres-container \
pg_dump -U username dbname > backup.sql

# PostgreSQL backup (all databases)
docker exec postgres-container \
pg_dumpall -U postgres > backup-all.sql

# PostgreSQL restore
docker exec -i postgres-container \
psql -U username dbname < backup.sql

# MySQL backup
docker exec mysql-container \
mysqldump -u root -p'password' dbname > backup.sql

# MySQL restore
docker exec -i mysql-container \
mysql -u root -p'password' dbname < backup.sql

# MongoDB backup
docker exec mongo-container \
mongodump --out /backup --gzip

# MongoDB restore
docker exec mongo-container \
mongorestore --gzip /backup

Volume-to-Volume Copy

# Create new volume
docker volume create my-volume-backup

# Copy data from one volume to another
docker run --rm -i -t \
-v my-volume:/from \
-v my-volume-backup:/to \
ubuntu bash -c "cd /from && cp -av . /to"

# Alternative: Use rsync for incremental copies
docker run --rm \
-v my-volume:/source:ro \
-v backup-volume:/dest \
ubuntu rsync -av /source/ /dest/

# Backup to AWS S3
docker run --rm \
-v my-volume:/data:ro \
-e AWS_ACCESS_KEY_ID=your_key \

Best Practices

Regular Schedule

Automate backups with cron or orchestration tools. Daily at minimum for production data.

Test Restores

Regularly test restore process. Backups are useless if you can't restore from them.

Off-Site Storage

Store backups on different server or cloud storage. Protect against host failure.

Retention Policy

Keep multiple backup versions. Follow 3-2-1 rule: 3 copies, 2 different media, 1 off-site.

Encryption

Encrypt sensitive data backups before upload. Use tools like gpg or openssl.

Monitoring

Monitor backup job success/failure. Alert on backup failures immediately.

Consistency Warning: Backing up live databases with tar may result in corrupted backups. Always prefer application-specific backup tools or stop containers during backup.
